# Claude Instructions for WorldCup2026

## Project Overview
World Cup 2026 prediction engine: Monte Carlo tournament simulator with Elo, Dixon-Coles, and bivariate-Poisson models. Real-time data ingestion from football-data.org, news APIs, and StatsBomb xG. Cloud-deployed to Cloudflare Workers + Pages.

## Key Architecture

### Directory Structure
```
run.py              # CLI entry point (argparse-based)
config.py           # Configuration constants
db.py               # Database / Neon PostgreSQL interface
requirements.txt    # Python dependencies

models/             # Core prediction engines
  elo.py            # Elo rating system
  poisson.py        # Dixon-Coles + bivariate-Poisson
  predict.py        # Match prediction interface
  tournament.py     # Monte Carlo simulation (50k runs)
  backtest.py       # Leakage-free accuracy testing
  variance.py       # QMC variance reduction
  tune.py           # Grid-tune hyperparameters
  calibrate.py      # Temperature scaling

sources/            # Data ingestion
  footballdata.py   # Live scores + historical results
  news.py           # News headline tagging
  statsbomb.py      # Expected goals (xG) data
  results.py        # Historical match parsing
  sportsdb.py       # SportsDB reference data

viz/                # Static site export + dashboard
  server.py         # Local development server
  export.py         # Build static HTML/JSON
  static/           # Frontend (app.js, graph.js, 3D)
  _shots/           # Shot maps

tools/              # Utilities
  audit.py          # Security / data quality gates
  cv_tactics.py     # OpenCV tactics detection (lazy-loaded)
  depgraph.py       # Dependency graph visualization

functions/          # Cloudflare Workers (Node.js)
  api/
    leaderboard.js  # Live leaderboard API
    run/
      submit.js     # Simulation submission endpoint

.github/workflows/  # GitHub Actions CI/CD
  full.yml          # Full pipeline (rebuild all)
  refresh.yml       # Lightweight live refresh
```

```bash

python run.py viz [port]                # Dev dashboard (default :8008)

```

## Key Models & Algorithms

### Ratings
- **Elo**: Traditional chess-style rating with K-factor decay
- **Dixon-Coles**: Poisson with time-weighted goal correlation
- **Bivariate-Poisson**: Shared covariance for home/away goals

### Simulation
- Monte Carlo: 50,000 tournament bracket samples (antithetic variance reduction)
- 2026 official draw baked into `field_2026.OFFICIAL_GROUPS`
- RPS (Ranked Probability Score) for leakage-free accuracy

### Data Pipeline
1. **Init**: Create/upgrade schema (idempotent, cold-start safe)
2. **Ingest**: Fetch historical + live + news + xG in parallel
3. **Train**: Recompute Elo → draw model → DC/BP marginals
4. **Simulate**: Monte Carlo with latest ratings
5. **Export**: Static JSON + HTML to `./dist` for CDN

## Code Conventions

### Imports
- Use absolute imports within package (e.g., `from models import elo`)
- Lazy imports in command handlers to keep startup fast
- UTF-8 encoding enforced at module entry (`sys.stdout.reconfigure`)

### Error Handling
- Return None or empty dict for graceful degradation
- Print error logs to stdout with timestamps `[HH:MM:SS]`
- Use `raise SystemExit(code)` for non-zero exits only in audit/cv commands

### Database
- `db.py` wraps Neon PostgreSQL via `psycopg`
- All tables created via `schema.sql` (version-controlled)
- Connections pooled; thread-safe within async loops

### Arguments
- Use `argparse` with subcommands (no positional-only args)
- Provide sensible defaults (e.g., `--year 2018`, `--runs config.SIM_RUNS`)
- Type hints on return values: `-> argparse.ArgumentParser`

## Performance Tuning

### Cold Start
- `config.SIM_RUNS = 50000` for accuracy; reduce for dev/testing
- `config.REFRESH_RUNS = 1500` for fast updates (25 min cadence)
- Lazy-load CV stack: `from tools import cv_tactics` only in `cmd_cv()`

### Data Freshness
- **Hourly**: News ingestion
- **3x daily**: Live score ingest + ratings + light resim (1500 runs)
- **Nightly**: Full rebuild (init → all sources → train → 50k sim → export)

### Variance Reduction
- **Antithetic sampling** in `Tournament.run(method="antithetic")`
- **QMC** (quasi-Monte Carlo) available in `models.variance`
- **Control variates** for goal differential

## Testing & Validation

### Unit Tests
- Run `python run.py audit` before deploy (no arguments)
- Checks: schema, data ranges, missing teams, encoding

### Backtests
- `python run.py backtest 2022 --refit 45` — accuracy from 2022 onward
- `python run.py backtest 2022 --compare` — BP vs DC head-to-head
- RPS is primary metric (lower = better)

### Load Testing
- `python run.py loadtest ./dist` — local static build
- `python run.py loadtest https://wcpa.pages.dev` — live site

## Cloud Deployment

### Cloudflare Workers
- `functions/api/leaderboard.js` — live leaderboard (KV cache)
- `functions/api/run/submit.js` — user simulation submission
- `wrangler.toml` — config (KV, D1, email bindings)

### Pages
- `python run.py export dist` → `npx wrangler pages deploy ./dist`
- Auto-deployed on git push or manual trigger

### Secrets
- `WORLDCUP_DATABASE_URL` — Neon connection string (prod DB)
- `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID` — Pages deploy auth
- `SPORTSDB_KEY` — SportsDB API key (optional; falls back to '3')

## Common Tasks

### Add a New Prediction Model
1. Create `models/new_model.py` with a fit/predict interface
2. Update `cmd_train()` to call `new_model.fit()`
3. Update `cmd_predict()` or `Predictor.predict()` to ensemble
4. Backtest: `python run.py backtest 2022`

### Ingest a New Data Source
1. Create `sources/new_source.py` with `ingest()` function
2. Add to `cmd_ingest()` conditional
3. Insert raw data into `DB` schema
4. Test: `python run.py ingest new_source` (or `python run.py ingest all`)

### Fix a Schema Issue
1. Edit `schema.sql`
2. Run `python run.py init` (idempotent, safe on warm DB)
3. Commit & deploy

### Debug a Simulation
- Enable verbose: `Tournament().run(verbose=True, runs=100)` (small N for speed)
- Check RPS: `python run.py backtest 2022` (leakage-free accuracy)
- Inspect ratings: `python run.py rankings 50`

## File Editing Best Practices

- **run.py**: Add new subcommands in `build_parser()` and corresponding `cmd_*()` function
- **models/**: Keep stateless; all state in DB
- **sources/**: Make idempotent (safe to re-run)
- **viz/**: Separate frontend (static/) from export logic (export.py)
- **schema.sql**: Version-control all schema changes; never delete columns
